# Command/event API boundary, with a non-blocking Sim state machine

We needed the exact same run to execute headless (tests, simulation, ghost-battle resolution) or under a UI, without the Sim depending on presentation or vice versa. We considered a shared mutable world state that presentation reads directly, and full event-sourcing (Sim only ever replays a persisted input log). We chose a command/event API instead: the Sim exposes commands (e.g. `sim_submit_captain_choice`) that mutate it and emits events (e.g. `DamageDealt`, `AwaitingCaptainDecision`) that presentation consumes — presentation never touches Sim internals. Shared state was rejected because presentation would have to diff state itself to know what changed; full event-sourcing was rejected as heavier machinery than a vertical slice needs.

The one captain decision per round meant the Sim needed to "pause" for input. We rejected blocking the Sim's own call stack via an OS thread or fiber (Odin has no native coroutines, so this would mean hand-rolled thread/fiber plumbing) in favor of a non-blocking state machine: a Tick advances the Sim until it needs input, emits `AwaitingCaptainDecision`, and returns; the driver must submit a command before the next Tick will progress. Calling Tick again while still awaiting a decision is a driver bug and `assert()`s rather than silently no-op-ing.

Ticks are instant and logic-only — a single Tick resolves an entire round and batch-emits all of that round's events, with no real-time throttling inside the Sim. This keeps sim time purely logical rather than tangled with frame timing, which is what deterministic replay (ghost battles) needs: a replay is just "feed the same seed and the same commands into Tick calls," independent of wall-clock timing. The Sim also owns its own explicit seeded RNG generator state (not the global default), so headless simulation and ghost-battle replay are bit-for-bit reproducible regardless of what else runs in the process.
